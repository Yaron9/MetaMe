# MetaMe 记忆系统升级方案：工程落地审查报告

## 1. 总体定调
原《记忆系统方案》提供了一个极具前瞻性的架构愿景，切中并准确指出了当前 MetaMe 在记忆管理上的痛点（如 `facts` 表达过载、检索污染）。

然而，**从个人单机 Agent 的敏捷落地和维护成本来看，原方案的部分设计过于理想化（Over-engineered）。** 本次审查剥离了其中不切实际的“企业级架构幻想”，保留了产生最大收益的核心机制，得出一套真正能跑、能快速落地的**收敛版（最优解）MVP 方案**。

## 2. 核心架构博弈与收敛方案

### 2.1 拒绝双写兼读，采取破釜沉舟的 ETL 迁移
* **原提议（妥协方案）**：新增 `memory_items`，老表 `facts` 与 `sessions` 继续作为兼容读源，查询 API 需同时处理两套 Schema。
* **工程隐患**：双读合并将带来极高的排序和分页负担，拖慢 `searchFacts` 的响应时间并产生冗长度极速叠加的代码技术债。
* **收敛解法**：**One-off ETL 迁移**。不写任何业务代码去兼容旧逻辑，直接编写一个 `migrate-v1-to-v2.js`，在系统升级时一次性清洗 `facts` 进入 `memory_items`。果断废弃老库旧读写逻辑，清爽上阵。

### 2.2 砍掉复杂的边表与图关系（`memory_edges`）
* **原提议（复杂架构）**：建立独立 `memory_edges` 表记录 `supersedes|derived_from|supports|contradicts`，构建知识溯源图谱。
* **工程隐患**：单机 AI 极少有递归查询关系谱图的诉求，FTS + SQL 拼装动态散列的图结构不仅会搞乱排序，还会极度消耗 LLM 的上下文。
* **收敛解法**：**单表自引用覆盖**。继续沿用并在 `memory_items` 内建 `supersedes_id`。如果遇到更新的约束，新记录直接取代旧记录（设旧记录为 archived）。“当前最新生效的规则是什么”永远比“它怎么从旧规则演化来的”重要。

### 2.3 摒弃人工审批流与候选表（`memory_candidates`）
* **原提议（沉重交互）**：设计独立的 `memory_candidates` 队列，强制进候选池等待合并程序提取或要求用户干预审批。
* **工程隐患**：频繁弹窗或单独审查候选池对于一位极客用户是极其反人类的重资产操作。
* **收敛解法**：**同表低置信度自净**。统一存入主表，初始标记 `state=candidate` 且低 `confidence`。搜索时该类记录自然权重垫底；一旦被低频触发命中且被 LLM 多次成功利用（累积 `search_count`），由夜间静默任务自动 Promote（晋升）为 `active` 成为高潜正式约束。

### 2.4 文件与 DB 强一致性的脆弱边界（`Working Memory` 脱库）
* **原提议（脆弱联席）**：`now/*.md` 作为文件视图，同时向 `memory_items(kind=working)` 持续进行双写索引以提供查询界面。
* **工程隐患**：用户在 VSCode 外部直接编辑 Markdown 将让 SQLite 索引立刻过时。此时唤醒 AI 将不可控地发生陈旧数据操作行为。
* **收敛解法**：**读时拦截，单一信源**。绝对不在 SQLite 里留存 `kind=working` 这种秒级更新的内容！`assembleContext` 被调用时如需“短期记忆”，底层直接改由 `fs.readFileSync` 读盘，确保无缓冲延迟。

## 3. 落地后的具体受益场景（UX/DX）

剥离掉过度设计的包袱后，这套最精简的收敛 MVP 将在以下日常高频编码场景中带来惊艳提升：

1. **无论会话多啰嗦，Agent 永远优先尊重“红线规矩”**
   通过 **Budget Assemble (分类预算组装)**，哪怕昨天除 Bug 产生了几十条无用的过程挣扎记录（Episodes），系统在拼接 Prompt 时能确保“该项目永远禁用某库”的约定（Convention）稳居并独占最上排份额。
2. **大脑不再“精神分裂”，告别左右互搏**
   利用**单表替代法则**。如果过去项目是 JS 开发，近期明确迁移到 TS 开发。当确定新的语言事实后，基于 `supersedes` 后者将直接斩断前者的生命周期。在接下来的任何召回中都绝对只输出 TS 的体系环境，Agent 永不含糊。
3. **工作台文件随手改，AI 瞬间同步不掉链子**
   通过**临时记忆脱库**，刚才在手边随手加上的 Markdown 计划项目，AI `dispatch` 之后可以零延迟毫秒读取，真正变成一个无缝衔接个人桌面的贴心“活”助理。
4. **后台静默成长的零打扰陪伴**
   丢掉审批流，意味着所有的学习都回归“暗流涌动”——从粗粒度知识放养式投递到主库，让时间与你的下意识调动频率作为大浪淘沙的考官，实现“无打扰自净”，系统变得就像正常人的肌肉记忆一般默默完善自身结构。

## 4. 实施路线建议

原本庞大的方案被优化掉“伪图”、“伪审批流”、“双轨写架构”之后，代码落地成本预期大幅缩减。
**工作量：2 ~ 3 个工作日：**
- **Day 1**: 创建 `memory_items` 表；撰写并一次性跑通 `migrate-v1-to-v2.js` 实施断代。
- **Day 2**: 构建全新的上下文分发核心 `assembleContext(query, budgets)`，并在其中挂载拦截针对 `now/` 下纯文本工作流的探针。
- **Day 3**: 将当前后台 `memory-extract.js` 和 `session-summarize.js` 的保存终点指向新 API，收尾。
